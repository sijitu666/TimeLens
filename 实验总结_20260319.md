## 三、核心问题分析

### 3.1 效果瓶颈分析

#### 问题1: RLVR提升缓慢，边际效益递减

**现象**: 从03-09到03-15进行了约5-6轮RLVR迭代，Charades R1@0.5从约59.4提升到约61.5，每轮提升仅约0.3-0.5点。

**根因分析**:
1. **奖励函数设计过于复杂**: EAR公式`R = R_answer × (1+α×R_evidence) × γ_efficiency + δ×𝟙(Perception)`包含4个超参(α, γ_coeff, δ, ref_steps)，优化空间大但调参困难
2. **证据奖励计算不稳定**: `R_evidence`基于提取的`[Evidence@...]`时间戳与GT的IoU，但模型生成的证据格式不稳定，导致奖励方差大
3. **Query分类不准确**: `rule_based`分类器仅基于关键词匹配，对复杂query的判断不够精确

**解决思路**:
1. 简化奖励函数，先只用`R_answer`作为基线，再逐步引入其他项
2. 增加格式奖励权重，强制模型按`\think...\n<answer>...`格式输出
3. 使用更精确的Query分类方法（如few-shot LLM分类）

#### 问题2: 与TimeLens-8B官方差距仍存在

**现象**: 当前最佳模型(Charades R1@0.5 ~61.5)与官方TimeLens-8B(63.0)仍有约1.5点差距。

**根因分析**:
1. **SFT数据质量**: 官方TimeLens使用了精心筛选的SFT数据，而EG-CoT数据生成依赖启发式规则，质量参差不齐
2. **训练数据量**: 官方模型可能在更多数据上进行了更长时间的训练
3. **Base模型差异**: 官方TimeLens-8B基于特定版本Qwen2-VL，而当前使用Qwen3-VL，可能存在架构/训练差异

**解决思路**:
1. 引入人工验证环节，筛选高质量EG-CoT样本
2. 使用外部LLM(OpenAI/Gemini)生成更准确的推理链
3. 尝试直接使用官方TimeLens-8B作为base模型进行RLVR

#### 问题3: 训练 pipeline 复杂度过高

**现象**: 从数据准备到模型训练再到评测，涉及约15个脚本/文件的协调，维护成本高。

**具体问题**:
1. **参数传递链过长**: `run_grpo_and_eval_qwen3_8b.sh` → `run_grpo_qwen3_8b.sh` → Python脚本，多层包装导致调试困难
2. **路径硬编码**: 部分脚本中仍存在硬编码路径，迁移到新环境需要多处修改
3. **错误处理不完善**: 中间步骤失败时，不会自动清理临时文件或给出明确错误提示

**解决思路**:
1. 使用Python编排工具(如Airflow/Prefect)或Makefile替代shell脚本链
2. 统一配置管理，使用YAML/JSON配置文件代替命令行参数
3. 增加中间检查点和自动恢复机制

### 3.2 设计思路反思

#### 反思1: EAR奖励函数的设计是否过于复杂？

**原始设计意图**: 通过引入`R_evidence`和`γ_efficiency`，鼓励模型生成与GT时间戳接近的证据，并避免冗余推理步骤。

**实际效果**: 
1. 由于证据提取依赖正则表达式，格式不稳定的回答会导致`R_evidence`计算不准确
2. `γ_efficiency`的超参`ref_steps`难以确定，不同复杂度query的"合理步数"差异大
3. 奖励方差大导致RL训练不稳定，需要更多steps才能收敛

**改进建议**:
1. 简化EAR为`R = R_answer + α * R_format`，先保证输出格式正确
2. 将证据奖励改为稀疏奖励（只在特定checkpoint评估）
3. 使用离线RL或DPO代替在线PPO/GRPO，减少奖励方差影响

#### 反思2: EG-CoT数据生成的启发式规则是否足够？

**当前方案**: 使用关键词匹配（如"why", "how", "reason"）判断query是否需要推理，并基于规则生成推理链。

**局限性**:
1. 自然语言的多样性导致关键词匹配准确率有限
2. 规则生成的推理链缺乏多样性，容易过拟合到特定模板
3. 未利用视频内容本身的信息（如场景变化、动作幅度）来判断复杂度

**改进建议**:
1. 使用少量人工标注数据训练Query分类器
2. 引入视觉信息（如帧间差异、光流）辅助复杂度估计
3. 使用LLM生成多样化推理链，并通过人工/自动过滤保证质量

#### 反思3: 多轮RLVR迭代的边际效益为何递减？

**观察**: 从第1轮到第5轮，每轮的提升幅度逐渐减小，最终趋于平稳。

**可能原因**:
1. **策略 collapse**: 模型可能收敛到局部最优，输出的多样性降低
2. **奖励 hacking**: 模型学会利用奖励函数的漏洞而非真正提升能力
3. **数据分布偏移**: 多轮训练后，训练数据分布与原始分布差异增大
4. **学习率/探索率**: 固定的学习率和探索策略无法适应不同阶段的优化需求

**改进建议**:
1. 引入多样性奖励，鼓励模型生成不同格式的回答
2. 使用population-based training或多模型ensemble，避免单一策略collapse
3. 定期重置策略网络（从SFT checkpoint重新初始化）
4. 动态调整学习率和KL penalty，早期鼓励探索，后期专注利用

## 六、技术实现要点回顾

### 6.1 EG-CoT (Evidence-Grounded Chain-of-Thought)

**核心设计**:
```
<think>
[Evidence@X.X-Y.Ys] 观察到的视觉证据描述
[Reasoning] 基于证据的推理过程
</think>
<answer>start - end</answer>
```

**关键实现**:
- `training/data/grounding.py`: 新增`PROMPT_EGCOT`模板，支持`--prompt_template egcot`开关
- `scripts/build_egcot_data.py`: 从TimeLens-100K构建EG-CoT训练数据，支持启发式/LLM生成
- 复杂度分类: 使用`QueryComplexityEstimator`区分Perception型vs Reasoning型Query

### 6.2 EAR (Evidence-Aware Reward)

**公式**:
```
R = R_answer × (1 + α × R_evidence) × γ_efficiency + δ × 𝟙(Perception)
```

**组件说明**:
- `R_answer`: 预测区间与GT的tIoU
- `R_evidence`: 提取的Evidence时间戳与GT的IoU+proximity
- `γ_efficiency`: 证据步数冗余惩罚（`ref_steps`控制期望步数）
- `δ`: Perception型Query不输出`<think>`的小奖励

**关键实现**:
- `training/train/reward_funcs.py`: 新增`evidence_aware_reward`函数，注册为`ear`
- `format_reward`放宽: 允许仅`<answer>...</answer>`（Perception型跳过推理）

### 6.3 Query复杂度估计

**两种模式**:
1. **rule_based** (默认): 基于关键词匹配（"why", "how", "reason"等）
2. **surprisal** (可选): 基于语言模型的困惑度，需要额外依赖

**用途**:
- EG-CoT数据构建: 区分Perception型vs Reasoning型Query，控制40/60分布
- EAR奖励: 对Perception型Query给予不同奖励
- 评测分组: 按Query类型分组统计指标

## 七、思路层面的核心问题

### 7.1 奖励函数设计：复杂度过高 vs 信号清晰度

**问题本质**: EAR奖励函数试图同时优化多个目标（答案准确性、证据质量、推理效率、Query类型适配），导致奖励信号复杂，方差大。

**反思**:
- 奖励函数应该简单、稀疏、与最终目标强相关
- 多目标优化应该通过课程学习或分层奖励实现，而非一次性加入
- 需要更系统地分析奖励函数各组件的梯度贡献

### 7.2 数据质量：启发式规则 vs 人工/LLM验证

**问题本质**: 当前EG-CoT数据生成主要依赖启发式规则，缺乏质量保证机制。

**反思**:
- 数据质量 > 数据数量，100条高质量样本可能比1000条低质量样本更有效
- 应该建立数据质量评估pipeline，包括人工抽检、LLM自动评估、训练动态监控
- 考虑引入主动学习，让人工标注集中在模型最困惑的样本上

### 7.3 训练策略：端到端RLVR vs 分阶段课程

**问题本质**: 当前采用端到端RLVR训练，模型需要在学习格式、学习推理、学习定位之间同时优化。

**反思**:
- 应该采用课程学习：先学格式→再学推理→最后精调定位
- 可以考虑使用DPO (Direct Preference Optimization) 替代PPO/GRPO，简化训练流程
- 需要建立更细致的中间指标监控（格式正确率、推理完整性、证据准确性）

### 7.4 评估指标：单一IoU vs 多维度质量

**问题本质**: 当前主要依赖R1@0.3/0.5/0.7和mIoU，但这些指标无法全面反映模型的推理能力。

**反思**:
- 应该引入更多维度的评估：格式正确率、推理链完整性、证据与答案一致性
- 需要人工抽检生成样本，分析错误类型分布（格式错误、证据缺失、推理跳跃、定位偏差）
- 考虑使用LLM作为评判者，评估推理链的合理性

## 十、技术实现要点回顾

### 10.1 EG-CoT (Evidence-Grounded Chain-of-Thought)

**核心设计**:
```
<think>
[Evidence@X.X-Y.Ys] 观察到的视觉证据描述
[Reasoning] 基于证据的推理过程
切除术
<answer>start - end</answer>
```

**关键实现**:
- `training/data/grounding.py`: 新增`PROMPT_EGCOT`模板，支持`--prompt_template egcot`开关
- `scripts/build_egcot_data.py`: 从TimeLens-100K构建EG-CoT训练数据，支持启发式/LLM生成
- 复杂度分类: 使用`QueryComplexityEstimator`区分Perception型vs Reasoning型Query

### 10.2 EAR (Evidence-Aware Reward)

**公式**:
```
R = R_answer × (1 + α × R_evidence) × γ_efficiency + δ × 𝟙(Perception)
```

**组件说明**:
- `R_answer`: 预测区间与GT的tIoU
- `R_evidence`: 提取的Evidence时间戳与GT的IoU+proximity
- `γ_efficiency`: 证据步数冗余惩罚（`ref_steps`控制期望步数）
- `δ`: Perception型Query不输出`<think>`的小奖励

**关键实现**:
- `training/train/reward_funcs.py`: 新增`evidence_aware_reward`函数，注册为`ear`
- `format_reward`放宽: 允许仅`<answer>...</answer>`（Perception型跳过推理）

### 10.3 Query复杂度估计

**两种模式**:
1. **rule_based** (默认): 基于关键词匹配（"why", "how", "reason"等）
2. **surprisal** (可选): 基于语言模型的困惑度，需要额外依赖

**用途**:
- EG-CoT数据构建: 区分Perception型vs Reasoning型Query，控制40/60分布
- EAR奖励: 对Perception型Query给予不同奖励
- 评测分组: 按Query类型分组统计指标

## 十一、关键经验总结

### 11.1 技术经验

1. **数据质量 > 模型复杂度**: 精心筛选的SFT数据比复杂的RL奖励函数更重要
2. **简单有效的奖励函数**: 奖励函数应该简单、稀疏、与最终目标强相关
3. **课程学习的重要性**: 端到端训练复杂任务，应该先学格式再学内容
4. **中间指标监控**: 除了最终指标，还需要监控格式正确率、推理完整性等中间指标

### 11.2 工程经验

1. **Pipeline复杂度控制**: 过多的脚本嵌套会增加调试难度，应该使用配置管理+简单编排
2. **路径管理**: 使用统一的配置管理路径，避免硬编码
3. **错误处理**: 增加中间检查点和自动恢复机制
4. **实验追踪**: 建立完善的实验记录和对比机制

### 11.3 研究思路

1. **基线意识**: 始终明确自己的方法相比基线的提升来源
2. **消融实验**: 每个组件的加入都应该有单独的消融实验验证
3. **失败分析**: 不仅要关注整体指标，还要深入分析失败样例
4. **迭代速度**: 快速迭代比一次性完美更重要
